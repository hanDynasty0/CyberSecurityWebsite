Denial Of Service Attacks

A Denial Of Service (DOS) attack is a cyber attack with the goal of making a software or hardware device unusable by essentially spamming requests on the device, causing an overload of its resources (e.g. memory, CPU, bandwidth, etc.). DOS attacks aren't used to breach data unless it is used as a way to prevent the target device's user from intervening while other malicious attacks are made on the target after the DOS attack.

How are DDOS different from DOS attacks?

DDOS stands for "Distributed Denial of Service". The objective of a DDOS attack is the same as that of a DOS attack: to render a software or hardware device unusable. However, the way DOS and DDOS attacks do this is slightly different. DOS attacks spam requests on the target device from a single device. DDOS attacks spam requests on the target device from multiple devices. DDOS attacks often use a network of multiple previously compromised or infected devices called "botnets". In other words, a DDOS attack is a hacker ganging up on a target alongside a bunch of zombies (botnet), whereas DOS attack is more one-on-one.

Example: Fork Bomb

A while ago, one of my friends asked me to test a piece of code. Here it is:

import java.io.File;
import java.io.FileWriter;
import java.io.IOException;
import java.awt.Desktop;
public class Main {
  public static void main(String[] args) throws IOException {
    File f = new File("robert.bat");
    f.createNewFile();
    FileWriter z = new FileWriter("robert.bat");
    z.write("@echo off\n%0|%0");
    z.close();
    Desktop dp = Desktop.getDesktop();
    dp.open(f);
  }
}

And yes, this is in Java.
So anyways I did in fact run this code on the Eclipse IDE, twice.
And it turned out it was a fork bomb.
Fork bombs are a unique form of DOS attack that has to be activated on the target's device, similar to a virus. The idea of a fork bomb is to quickly and continually create new files. This happens because each file created by a fork bomb then duplicates, and create more files that also duplicate, meaning that the number of files created on your device will grow exponentially. All of this activity consumes your device's memory and requires more and more CPU processing power to create files until the device crashes. Pressing Ctrl+C won't stop the fork bomb, and even if the code is terminated on the IDE it was run on, the device will continue to be super slow.

(Above: Visual representing the duplication process of a fork bomb)

However, fork bombs are pretty harmless. They don't do permanent damage to a device, and all you would have to do to restore the device's resources is to force your device to shut down and reboot. So you can try running the above Java code yourself to see a DOS attack in action (at your own risk).
There are ways to prevent fork bombs altogether. An installed antivirus should be able to advise caution to a user before running a fork bomb. On computers that run on Linux or Unix, a user can enter the command "ulimit-u" or "ulimit-a" to see the number of processes that the device's operating system can run, and then edit that number using the command "ulimit -$ -u (your number of choice)" in order to lower the number and prevent too many files from being created when a fork bomb is detonated. Unfortunately, since fork bombs seem to be easier to implement on Linux or Unix systems, I could not find any more ways to prevent fork bombs on other systems such as Windows, although the fork bomb code I have provided also works on Windows devices (I did try).
